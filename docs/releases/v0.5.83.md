# v0.5.83 - Phase 43 Message-Level File Attachment APIs

This release upgrades shared files from generic message attachments into fully hydrated message-level objects. The frontend can now render a file card inline in the channel feed or thread without extra stitching logic.

## Added

- `GET /api/v1/messages/:id/files`
  - Returns hydrated file attachments scoped to a single message.
  - Intended for lazy loading, message detail drawers, and richer attachment inspectors.
- Rich `metadata.attachments` file payloads on `GET /api/v1/messages`, `GET /api/v1/messages/:id/thread`, and `POST /api/v1/messages`
  - `preview_kind`, `preview_url`, `download_url`
  - `uploader`
  - `comment_count`, `share_count`, `starred`
  - `knowledge_state`, `source_kind`, `summary`, `tags`
  - `is_archived`, `retention_days`, `expires_at`
  - nested `file` and `preview` objects

## Changed

- Newly created IDs now use prefixed UUIDs for:
  - channel messages
  - share-generated file messages
  - DM conversations
  - DM messages
  - workspace invites
  - agents

## Why This Matters

- Windsurf can render shared files as first-class cards in conversation, not just links.
- File knowledge metadata now travels with the message object itself, which is a better base for later wiki and LLM retrieval surfaces.
- Remaining timestamp-style string IDs continue shrinking, so URLs and client-side state become more consistent.

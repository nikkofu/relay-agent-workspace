# v0.5.92 - Phase 52 Digest Automation And Knowledge Inbox APIs

## Summary

This release turns Relay's channel knowledge digest into an ambient delivery system instead of a manual-only action. The backend now supports:

- persisted digest schedules per channel
- automatic digest publishing on a timezone-aware schedule
- a cross-channel knowledge inbox
- Home-level digest aggregation for badges and recent digest cards
- realtime digest publish notifications for UI invalidation

## Added APIs

### Digest Schedule

- `GET /api/v1/channels/:id/knowledge/digest/schedule`
- `PUT /api/v1/channels/:id/knowledge/digest/schedule`
- `DELETE /api/v1/channels/:id/knowledge/digest/schedule`
- Schedule fields:
  - `window`
  - `timezone`
  - `day_of_week`
  - `day_of_month`
  - `hour`
  - `minute`
  - `limit`
  - `pin`
  - `is_enabled`
  - `last_published_at`
  - `next_run_at`

### Knowledge Inbox

- `GET /api/v1/knowledge/inbox`
- Query params:
  - `scope` = `all` | `starred`
  - `limit`
- Response fields per inbox item:
  - `id`
  - `channel`
  - `message`
  - `digest`
  - `is_read`
  - `occurred_at`

## Realtime Contract

- New websocket event:
  - `knowledge.digest.published`

This event is emitted for both manual and scheduled digest publishes.

## Home Contract Additions

- `GET /api/v1/home` now also returns:
  - `knowledge_inbox_count`
  - `recent_knowledge_digests`

This is intentionally designed so the Home surface can render digest badges and cards without building a second inbox-specific aggregation flow in the frontend.

## Windsurf Handoff

Please wire the following next:

1. Digest schedule controls using `GET|PUT|DELETE /api/v1/channels/:id/knowledge/digest/schedule`
2. A top-level `/workspace/knowledge/inbox` route using `GET /api/v1/knowledge/inbox`
3. Home dashboard digest cards and badges using `knowledge_inbox_count` and `recent_knowledge_digests`
4. Realtime invalidation of digest banners/inbox surfaces on `knowledge.digest.published`

Compatibility note:

- Knowledge inbox items intentionally reuse stable notification IDs like `knowledge-digest-<message_id>`, so the existing `POST /api/v1/notifications/read` path can be reused for read-state changes.
